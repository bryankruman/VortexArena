# Upstream Watch — tracking original-Xonotic contributions

This directory is the **single source of truth** for what the upstream Xonotic project is doing
*right now* and whether any of it should flow into **Vortex Arena** (the game formerly codenamed
XonoticGodot). It is the forward-looking companion to [`../parity/`](../parity/): parity measures how
faithfully we reproduce a **pinned** Base revision; upstream-watch tracks what has changed **since**
that pin and turns each new upstream contribution into an explicit *port / reject / defer* decision.

It exists so that "is upstream doing anything we want?" is a **repeatable weekly routine** with a
durable paper trail, not an occasional manual scan that silently misses things.

---

## 1. Scope — what we watch

| Source | Local path | Remote | Main branch | What it holds |
|---|---|---|---|---|
| **`data`** | `../../../Base/data/xonotic-data.pk3dir` | `gitlab.com/xonotic/xonotic-data.pk3dir` | `master` | QuakeC gameplay (`qcsrc/`), cfgs, effectinfo, shaders, map/model data — **most of what we port** |
| **`dp`** | `../../../Base/darkplaces` | `gitlab.com/xonotic/darkplaces` | `master` | The reference **engine** — much of it is replaced by Godot, so most engine commits are `n/a`, but protocol/physics/collision/asset-format changes can still matter |

**Two harvest streams, both since `2026-06-01` (the cutoff, configurable):**

1. **Every commit that lands on `master`** — the accepted, merged work.
2. **Every *active* branch that is ahead of `master`** — proposed/in-flight contributions. The
   GitLab MR API is queried for the *open-MR* signal (title, author, url); open MRs are **always**
   surfaced. Branches without an MR are surfaced only if recently active.

   > **Why a recency gate.** The two repos carry **~450 branches ahead of master going back to the
   > year 2000** — GitLab branches have no open/closed state, so the overwhelming majority are
   > long-abandoned experiments, not live contributions. The harvester hides branches whose last
   > commit predates a lookback window (default 12 months) *and* have no open MR — but it **counts
   > and reports** how many it hid (never a silent drop), and `--all-branches` shows everything.
   > A branch we already triaged is always re-surfaced when it gets new commits, regardless of age.

Nothing is dropped silently. High-noise commits (Transifex/translation autosync, pure CI/build
chores) are **flagged** as low-priority in the worklist, not filtered out, so the ledger stays
honest about coverage — the same "no silent caps" discipline the parity system uses.

> **Pin awareness.** Our parity baseline is `v0.8.6-1779-g863cd3e84` (data submodule). Upstream-watch
> deliberately looks at everything *after* that. When an accepted item is ported, note whether it
> also moves our effective baseline for the touched subsystem (link the parity registry unit).

## 2. Artifacts in this directory

| Path | What |
|---|---|
| `README.md` | This document — the process, rubric, vocab, and how to run it. |
| `LEDGER.md` | The running index: one row per triaged contribution — `UW-####` id, source key, port-worthiness, decision, one-line summary. **The at-a-glance answer to "what have we decided?"** |
| `items/UW-####-<slug>.md` | Per-contribution deep-dive analysis (only for items worth more than a ledger line). |
| `items/_TEMPLATE.md` | The deep-dive template. |
| `_inbox/worklist-<date>.md` / `.json` | Generated by the harvester: the un-triaged candidates for one weekly run. `.md` for humans, `.json` for the analysis workflow. Ephemeral; safe to delete after triage. |

## 3. The identifier scheme

Every tracked contribution gets a **stable, unique, never-reused id**: `UW-####` (Upstream Watch),
zero-padded, assigned sequentially at triage time (next id = highest in `LEDGER.md` + 1).

Each row also carries a **source key** — the machine-checkable dedup handle the harvester uses so the
same commit/branch is never re-triaged:

- **master commit:** `{repo}@{short}` — e.g. `data@a1b2c3d4e5f6`
- **branch / MR:** `{repo}:{branch}@{tip}` — e.g. `data:mario/vehicles-rework@0f1e2d3c4b5a`
  (the tip hash lets the harvester re-surface a branch when it gets new commits after we triaged it).

> **`UW-####` is permanent.** Like parity `id`s, never rename or recycle one — it may be cited from
> `TODO.md`, a parity registry row, a branch name, or a commit message.

## 4. Port-decision vocabulary (the `Decision` column)

| Value | Meaning |
|---|---|
| `pending` | **Default.** Analyzed, but the port/reject call is Bryan's to make. |
| `port` | Accepted — will be ported as-is. Spawns a `TODO.md` entry and (usually) a parity registry row. |
| `adapt` | Accepted, but needs modification to fit Vortex Arena (design change, C#/Godot idiom, our netcode). |
| `ported` | Done and merged into our tree (link the commit/PR). |
| `defer` | Worth doing later, not now — blocked, depends on unfinished upstream work, or off the current roadmap. |
| `reject` | Won't port. **Rationale required** (out of scope, conflicts with design goals, engine-internal, superseded). |
| `n/a` | Not applicable to the port — DarkPlaces internals we've replaced, translations, upstream CI/packaging. |

Port-worthiness (the ledger's **Value** column) is a separate at-a-glance rollup (auditor's
recommendation, independent of the final human decision): **🟢 high · 🟡 medium · 🟠 low · 🔴 none**.

## 5. The analysis rubric

Each candidate is scored on these dimensions (captured in the ledger one-liner for small items, or a
full `items/UW-####-*.md` doc for anything non-trivial). This is the "is it worth considering, what
does it do, how portable, how complete, how good, does it fit us" that the routine must answer:

1. **What it does / how it works** — the feature or fix in one paragraph, from reading the diff (not
   just the commit title). Name the Base symbols/files it touches.
2. **Portability** — which layer it lives in and how hard it is to bring across:
   - `qc-gameplay` — QuakeC in `qcsrc/`: usually portable to `src/XonoticGodot.Common/**` (our bread and butter).
   - `data-cfg` — cfg/balance/effectinfo/notification data: often a near-mechanical value port.
   - `asset` — models/maps/textures/sounds: portable if our loaders support the format.
   - `dp-engine` — DarkPlaces C: mostly `n/a` (Godot replaces it) **unless** it's protocol/physics/
     collision/asset-format behavior we reimplement — then it's a spec change we must mirror.
   - `build/i18n/ci` — almost always `n/a`.
3. **Completeness (upstream)** — is it merged to master, an open MR under review, or a stale WIP
   branch? Does it look finished, or half-built/experimental?
4. **Quality** — implementation soundness as read: is it a clean fix, a hack, does it carry tests,
   does it match how the surrounding Base code is written?
5. **Roadmap / design alignment** — does it serve Vortex Arena's goals, or is it upstream-specific
   churn (translations, their CI, cosmetic upstream choices we've intentionally diverged from)?
   Cross-check `planning/` roadmap docs and existing `intended_divergence` decisions.
6. **Effort & risk** — rough size (S/M/L) and what it interacts with (netcode, determinism, hot
   paths, our existing divergences). Flag anything that could regress a fixed bug (see the memory
   index / `planning/*` postmortems).

**Rollup:** the auditor sets `Port-worthiness` from the above and proposes a `Decision` (usually
`pending` for anything with real gameplay impact — those are Bryan's call).

## 6. The weekly routine

```
        harvest  ──▶  triage  ──▶  analyze  ──▶  decide  ──▶  integrate
      (mechanical)  (dedup +    (rubric per   (Bryan sets   (accepted → TODO
                     classify)   candidate)    Decision)     + parity row)
```

1. **Harvest** *(mechanical, ~1 min)* — run the harvester. It fetches both repos, enumerates new
   master commits + branches-ahead + open MRs since the cutoff, drops anything already in
   `LEDGER.md`, and writes `_inbox/worklist-<date>.md` + `.json`.

   ```bash
   python tools/upstream-watch.py            # default cutoff 2026-06-01, fetches origin
   python tools/upstream-watch.py --no-fetch # offline: use local refs as-is
   ```

2. **Triage** *(fast)* — skim the worklist. Noise items (i18n/CI/chore, pre-flagged) get a one-line
   ledger row with `Decision: n/a` and no deep-dive. Everything else goes to analysis.

3. **Analyze** *(the substance)* — run the analysis pass over the worklist. It assigns `UW-####`
   ids, applies the rubric per candidate, writes ledger rows, and creates `items/UW-####-*.md` for
   non-trivial ones. Two ways to run it:

   ```
   # (a) Deterministic multi-agent workflow (one analyst per candidate, then a scribe writes rows):
   Workflow { name: "upstream-watch", args: { worklist: "planning/upstream-watch/_inbox/worklist-<date>.json" } }

   # (b) No-workflow fallback: hand the worklist to Claude in chat —
   #     "Analyze planning/upstream-watch/_inbox/worklist-<date>.md per the upstream-watch rubric,
   #      assign UW ids, and append ledger rows." Good for a light week (few candidates).
   ```

4. **Decide** *(Bryan)* — review the new `pending` rows in `LEDGER.md`, set each `Decision`
   (`port` / `adapt` / `defer` / `reject` / `n/a`). This is the one human-in-the-loop step.

5. **Integrate** — for each `port` / `adapt`: open a `TODO.md` entry (cite the `UW-####`), and, when
   it's a parity-relevant behavior, add/adjust the `planning/parity/registry/<unit>.yaml` row so the
   new upstream behavior becomes part of the tracked spec. Flip the ledger `Decision` to `ported`
   with the commit link when done.

## 7. Cadence & automation

The routine is designed to be **cheap to run weekly**. Options, lightest first:

- **Manual** — run the harvester + analysis each Monday. Zero standing infrastructure.
- **Scheduled local run** — a `/loop 1w` or the `schedule` skill kicks the harvester + analysis and
  leaves fresh `pending` rows for Bryan to decide. (Harvester needs network + the local Base repos.)
- **Scheduled cloud agent** — a routine that runs the harvest + analysis and pings when new
  `high`/`medium` port-worthy items land. Best if the Base repos are reachable from the runner.

Whichever is chosen, **the decision step stays manual** — the machine surfaces and recommends; the
port/reject call is Bryan's.

---

## 8. Relationship to the rest of `planning/`

- **`parity/`** — once an item is `ported`, the behavior it adds/changes should be reflected in the
  parity registry (new row, or a note that our effective baseline for that unit moved past the pin).
  Upstream-watch is "what's new upstream"; parity is "how faithful are we to what we've adopted."
- **`TODO.md` / `FIXME.md`** — accepted items become tasks there, citing the `UW-####`.
- **`WISHLIST.md`** — `defer` items with no near-term slot can be cross-linked here.
- **Memory index** — before accepting anything that touches movement, netcode, warpzones, spawns,
  particles, or perf hot paths, check the auto-memory postmortems so a port doesn't reopen a
  bug we already closed.

## 9. Suggestions / possible enhancements

These go beyond the ask; adopt any that earn their keep:

1. **RSS/Atom instead of full fetch** — GitLab exposes `…/-/commits/master?format=atom` and an MR
   feed. A future harvester mode could poll those to detect "anything new?" without a full clone
   fetch, useful for a cloud runner that doesn't hold the Base repos.
2. **Auto-link to parity units** — have the analysis step map each candidate's touched `qcsrc/`
   files to the parity registry unit(s) that cite them, so "does this regress/extend something we
   track?" is answered automatically. The `port_refs`/`base_refs` in the registry already carry the
   file→unit mapping.
3. **Diff-against-our-tree** — for `qc-gameplay` items, a bot pass could check whether the touched
   Base symbol is already ported in `src/`, so the ledger says "we have this file — this is a *delta*
   to an existing port" vs "brand-new subsystem." Sharpens effort estimates.
4. **Security/CVE lane** — DarkPlaces and the data loaders parse untrusted network + asset data. Flag
   any upstream commit whose subject/paths smell like a memory-safety or parser fix (`overflow`,
   `bounds`, `sanitize`, `CVE`, `heap`, `oob`) as **must-review**, even when the engine is otherwise
   `n/a` — those often mirror a bug in our reimplementation.
5. **Contributor watch** — track a short list of high-signal upstream authors (core devs) so their
   branches get surfaced even before an MR is opened.
6. **Quarterly re-pin review** — periodically ask "should we advance the parity pin to a newer Base
   tag wholesale?" using the accumulated ledger as the evidence base.
7. **`--since` drift guard** — the harvester records the last successful run date; a future version
   could auto-advance the effective cutoff so a missed week never leaves a gap (it re-scans from the
   last run, not a fixed date).
